## Theory
Lexical analysis is the very first phase in the pipeline of a compiler or interpreter. Its primary purpose is to read the raw source code—which is just a continuous sequence of characters—and convert it into a structured sequence of logical units called **tokens**. 

This process is handled by a program usually called a **Lexer**, **Scanner**, or **Tokenizer**. A lexer groups individual characters (lexemes) together based on the rules of the language and assigns them a categorical meaning (Token Type). For example, the string `10.5` becomes a `FLOAT` token, while `x` becomes an `IDENTIFIER` token. 

Separating lexical analysis from syntax parsing offers significant benefits, including simplifying the parser's logic, enhancing efficiency, and providing cleaner compiler architectures.

### 2. Lexer Core Logic (`Lexer.py`)

### 3. Digits, Strings and Strings Scopes
I included specific logic for capturing extended arrays of characters:
- **Numbers**: Function `read_number()` checks continuously if upcoming characters are part of a continuous digit structure. It flags internal decimation `.`. If isolated correctly, it flags the resultant as `FLOAT` token; otherwise, it resolves an `INTEGER`.
- **Strings**: It checks for an opening quote `"`, locks into `read_string()`, ignores keywords entirely reading blindly until the closing quote `"` finishes the boundary.

### 4. Line and Column Tracking
Rather than just grabbing keywords globally, this lexer faithfully reconstructs the physical architecture. It uses `self.line` and `self.column`. This helps trace syntax error possibilities during actual language processing (acting as debugging coordinates).
